Technical Trial Objective

You are being asked by a customer to create an Air Ticket Reservation system. Create the Architecture and Design of the system. And then implement the framework of the system.

Functional Requirements

Here is a summary of the functional requirements for the system

  1. Public users

    1. Are able to login using their google+, twitter or LinkedIn accounts. Either one is enough.

    2. Are able to reserve air tickets searching from multiple flights based on multiple criteria.

    3. Can make payment using credit cards.

    4. Receive tickets as PDF via email. This email should contain useful links like “My Bookings”, “Check-In Now”, “Cancel booking” that will not require the user to login.

    5. Are able to perform online check-in/cancellation, from 48 hrs and upto 4 hrs from departure, choose seats, and print boarding passes.

    6. Receive email alert at 48 hrs from departure, regarding their reservation, to encourage them for early online check-in. This email should contain a “Check-In Now” link that will not require the user to login.

  2. Airline staff

    1. Are able to login using their internal credentials, even from public computers.

    2. Receive reservation chart as PDF at 1 hrs from departure.

    3. Cancel any ticket, generating email alert and refund to the public user.

    4. Cancel the whole flight, generating email alerts and refunds to the public users.

Assume any other functionality that may be necessary to achieve the above requirements based on logic and your experience. In case you think there are details missing, think of this as skill assessment, where you can assume details on your own.

Other Technical and Non-functional Requirements

  1. Must create a Single-page Application (SPA) based web portal with responsive design.

  2. System core must be ready to serve mobile apps in future w/o any modification.

  3. System will be installed on cloud with services that you recommend enabled.

  4. All the alerts sent should be guaranteed (not lost due to some technical errors).

  5. Technologies are not recommended as it is a design decision you should make as per the position applied for.

Scope of work for the “Framework”

Once you have developed the architecture and design of this system, you need to implement/code a framework. Here is the scope of work for this framework

  1. Create solution/projects with 3rd party references/packages integrated

  2. Create layers, modules, technical / cross cutting concern classes

  3. All integrations with 3rd party libraries/references should be fully coded

  4. Security and authentication should be fully coded

  5. Complete and test one sample “functional chain” end to end, so for example one CRUD operation should work end to end from database to the UI.

  6. Write automatic integration/unit tests with at least 70% coverage for the “functional chain”, to demonstrate you can write tests for different layers.

If you are still confused about the scope of implementation, answer this question

  1. As a system/application architect who can do low level design and code also, how much framework you need to implement so that junior developers can just copy paste and develop the whole system functionally without any technical research?

What to deliver?

Design Document

Create a single design document containing the following:

  1. High level requirement analysis

  2. High level presentation of the data model

  3. Architecture diagrams describing the composition and working of the system, explaining the component interaction and process, control and data flows.

  4. Explain the breakdown of the system into components with technical implementation details of each component, along with the design patterns involved (with reasons that justify your choices).

  5. Present the UI screens and how they are connected through user interactions with those screens

  6. Use both visual elements (diagrams) and text descriptions to maximize the amount of information conveyed, while keeping the document as compact as possible.

  7. Diagrams in any format (like UML) are not mandatory but nice to have, it is upto you how best to depict the logical design.

  8. Create the document as a single office file (doc, pdf, etc), or a set of linked files where only one needs to be opened (index.html, etc).

Readme Document

Create a text file with the following information

  1. Instructions to install and configure prerequisites or dependencies

  2. Instructions to create and initialize the database

  3. Assumptions you have made (a good quality response will include your thought process and assumptions)

  4. Requirements that you have not covered in your implementation

  5. Instructions to configure and prepare the source code to build and run properly

  6. Issues you have faced while completing the assignment, if any

  7. Any feedback you wish to give for improving the assignment and our process.

Source Code

You should deliver the implemented source code, excluding any dependencies. For the dependencies the readme file should have proper instructions on how to download and install them.

What we will evaluate?

Readme/Design Documents

  1. The quality and completeness of the documentation

  2. The functional coverage provided by the system design w.r.t the specifications

  3. The quality, depth and novelty of the technical solutions proposed for implementing each component

  4. Technology choices made, design patterns applied and the rationale given

  5. Assumptions about the missing requirements which may not be described in the specifications and feedback about the same

  6. Clarity of the code dependencies, configuration, build, and run steps

  7. The completeness of these documents is as important as that of the code. If they are not complete, the code will not be evaluated. So don’t leave to the end with only a couple of hours remaining.

Framework Code

  1. Specs : The completeness of the implementation as per the requirements and scope

  2. Design : The quality, depth and novelty of the solutions, technical choices and integrations

  3. Code : The quality of the hand written code and its documentation

  2. Any deliverable except for what are specified.
